## Problem Statement

The go-guardian MCP server's knowledge base (patterns, anti-patterns, OWASP findings, vulnerabilities, scan history, session findings, Renovate preferences) is entirely invisible during normal operation. There is no way to see which agents are calling which tools, how often patterns are being used for prevention, whether the knowledge base is growing or stagnating, or to curate its contents. Users must resort to raw SQLite queries to understand what the learning engine has captured.

## Solution

An embedded admin UI served directly by the go-guardian-mcp binary as a concurrent HTTP server. The UI provides 8 views covering full observability (dashboard, agent activity, scan trends, session findings) and full management (pattern CRUD with audit trail, quality suggestions). Built as a Preact + TypeScript SPA, embedded via Go's `embed.FS`, shipping as part of the single binary with zero additional dependencies.

## User Stories

1. As a developer, I want a dashboard showing total patterns, recent learning events, scan health, and active session info at a glance, so that I can quickly assess the state of my knowledge base.

2. As a developer, I want to browse, search, edit, and delete lint patterns and anti-patterns with filters by rule, source, and frequency, so that I can curate the knowledge base and remove low-quality entries.

3. As a developer, I want to see scan history and trend charts (improving/degrading/stable) over time per scan type, so that I can track whether code quality is getting better or worse.

4. As a developer, I want to view current and past session findings with filters by agent, severity, and finding type, so that I can understand what each agent flagged during a session.

5. As a developer, I want to browse OWASP findings by category (A01-A10) with frequency and file pattern data, so that I can identify recurring security issues.

6. As a developer, I want to browse the vulnerability cache showing all cached CVEs per module with severity and fix versions, plus dependency decision history, so that I can track my dependency risk posture.

7. As a developer, I want to see Renovate preferences, rules, and config scores with suggestion acceptance rates, so that I can understand how my Renovate config is evolving.

8. As a developer, I want to see a real-time agent activity log showing which MCP tools were called, by which agent, with what parameters and duration, so that I can understand how agents use the knowledge base.

9. As a developer, I want the admin UI to suggest pattern quality improvements (flag empty do_code, very low frequency, duplicate rules) and recommend merging, updating, or removing them, so that the knowledge base stays healthy.

10. As a developer, I want the dashboard to auto-refresh every 5-10 seconds via polling, so that I see live updates without manual page reloads.

11. As a developer, I want deleted patterns to be soft-deleted with a history table, so that I can restore accidentally removed patterns.

12. As a developer, I want the admin UI to start automatically when `GO_GUARDIAN_ADMIN_PORT` is set, bound to localhost only, so that it's secure by default and requires no auth.

## Implementation Decisions

- **Embedded in MCP binary**: The admin HTTP server runs as a goroutine in `main.go` when `GO_GUARDIAN_ADMIN_PORT` env var is set. Shares the same `db.Store` instance. No separate process.
- **Frontend stack**: Preact + TypeScript SPA. Built with Vite or esbuild. Static assets embedded via Go's `embed.FS` directive. Ships in the single binary.
- **Port configuration**: `GO_GUARDIAN_ADMIN_PORT` env var only (no CLI flag). Default: disabled (unset). When set, HTTP server binds to `127.0.0.1:<port>`.
- **No authentication**: Localhost-only binding. Same trust model as the MCP server itself.
- **Charts**: uPlot (~30KB) for time-series trend visualizations in the scan trends view.
- **Auto-refresh**: Dashboard and activity log poll the JSON API every 5-10 seconds for updated data.
- **New schema: `mcp_requests` table**: Logs every MCP tool invocation with timestamp, tool_name, agent (from session context), params summary (truncated), duration_ms. Auto-prune entries older than 7 days (cleanup on session start).
- **New schema: `pattern_history` table**: Records edits and soft-deletes with before/after snapshots. Enables undo/restore from the admin UI.
- **Soft delete for patterns**: Add `deleted_at` column to `lint_patterns` and `anti_patterns`. Existing queries filter on `deleted_at IS NULL`. Admin UI can restore by clearing `deleted_at`.
- **Pattern quality suggestions**: Server-side analysis that flags patterns with empty `do_code`, frequency of 1, or duplicate `dont_code` across different rules. Returned as a JSON endpoint consumed by the dashboard.
- **REST API**: All views backed by JSON endpoints under `/api/v1/` (e.g., `/api/v1/patterns`, `/api/v1/trends`, `/api/v1/activity`). SPA fetches from these.
- **8 views**: Dashboard, Patterns browser, Scan trends, Session findings, OWASP findings, Vulnerability cache, Renovate insights, Agent activity log.

## Testing Decisions

- **API handler tests**: Table-driven tests for each `/api/v1/*` endpoint. Use `httptest.NewServer` with a real `db.Store` (in-memory SQLite). Test CRUD operations, filters, pagination, soft-delete/restore.
- **MCP request logging tests**: Verify middleware logs calls correctly, truncates params, prunes old entries.
- **Pattern quality suggestion tests**: Seed patterns with known quality issues, verify suggestion engine flags them correctly.
- **Prior art**: Existing tests in `tools/*_test.go` use the same pattern — create a temp DB, seed data, assert behavior. Follow that convention.
- **Frontend**: No unit tests for the SPA initially. Backend API tests provide coverage. Frontend testing can be added later if needed.

## Out of Scope

- WebSocket/SSE for real-time push updates (polling is sufficient for v1)
- User authentication or multi-user access control
- Remote access (always localhost-only)
- Frontend unit/integration tests
- Coverage gap analysis ("you have handler files but no handler patterns")
- Export/import of patterns
- Multi-database support (one DB per MCP server instance)

## Further Notes

- The `session-start.sh` hook should persist `GO_GUARDIAN_ADMIN_PORT` via `CLAUDE_ENV_FILE` so the admin UI starts automatically in Claude Code sessions when configured.
- The admin UI port should be logged on startup alongside the existing "go-guardian MCP server started" message.
- Consider adding the admin UI URL to the `/go-doctor` healthcheck output.

## Deferred Ideas

- Coverage gap analysis: scan codebase file types vs pattern file_globs to suggest areas lacking patterns
- Pattern import/export (JSON/YAML) for sharing knowledge bases between projects
- WebSocket live updates for the activity log
- Dark mode / theme customization
- Multi-project view (aggregate data from multiple guardian.db instances)
